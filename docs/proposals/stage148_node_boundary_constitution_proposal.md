# Stage148 Proposal - Node Boundary Constitution

Stage147 created a stable manifest packet body, but the repository still needed an explicit constitutional statement of which packets each node may consume. Stage148 fills that gap by turning Node1, Node2, and Node3 access into auditable evidence.

## Stage148 Must

1. Bind Stage146 state contracts and Stage147 manifest packets to explicit Node1, Node2, and Node3 access levels.
2. Publish deterministic route maps that explain how packets flow into planning, rendering, and criticism.
3. Preserve Node2 surface-only access and keep raw reveal access at zero.
4. Prove provider-zero, write-zero, and reveal-boundary invariants still hold.
5. Signal that Stage149 can close Page01 with a release gate rather than more contract invention.

## Required Invariants

- Provider calls remain disabled.
- Runtime training remains disabled.
- Model weight updates remain disabled.
- Automatic memory writes remain disabled.
- Automatic canon mutation remains disabled.
- Automatic repair apply remains disabled.
- Node2 may consume only surface-safe projections.
- Node3 may receive critic summaries but not mutation authority.

## Out of Scope

- New narrative state contracts
- New manifest loaders
- Live provider execution
- Memory storage or retrieval
- Runtime mutation
- Stage149 release promotion logic itself
