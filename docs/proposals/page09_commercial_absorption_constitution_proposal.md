## 0. Purpose

Page09 is the first page after the Page08 lineage authority layer. Its purpose is to define how commercial writing-tool concepts and the uploaded Codex / Narrative IDE proposal may enter V1700 without weakening inherited logic.

Page09 does not implement Story Repository, EAT8D extraction, Story Patch, Narrative PR, Narrative IDE, MultiWork, plugin, or learning. It creates the constitution that decides how such features will later be admitted.

Page09 establishes four authority artifacts:

```text
1. Commercial Feature Taxonomy
2. Feature-to-Contract Mapping
3. Provider Sandbox / Writer Approval Policy
4. Page09 Release Seal
```


## 3. Expert review roles

This proposal is written through three expert viewpoints:

```text
Chief Principal Architect
Chief Principal Compiler Engineer
Chief System Principal Engineer
```

The architect defines system shape. The compiler engineer defines machine-readable contracts. The system principal engineer governs risk, release boundaries, and next evolution direction.

## 4. Chief Principal Architect review

### 4.1 Architect position

The architect argues that Page09 is the translation layer between external product inspiration and V1700 internal authority. Commercial systems are useful, but they must not become direct imports.

The system must convert external features into native V1700 structures:

```text
feature -> contract -> evidence schema -> fixture -> gate -> report -> human review where needed -> release seal
```

### 4.2 Architect risk

The largest risk is product mimicry. If V1700 directly copies commercial features, it may import hidden assumptions:

```text
automatic generation as manuscript authority
context preview without reveal boundary
AI rewrite without author approval
collaboration without permission model
personalization without audit
plugin without capability scope
```

### 4.3 Architect requirement

Page09 must classify all commercial feature families before Page10 begins. Each family must declare whether it is:

```text
CORE_ROADMAP
GATED_CAPABILITY
ADVISORY_SURFACE
PRODUCT_SURFACE
FUTURE_EXTENSION
REJECTED_OR_DEFERRED
```

### 4.4 Architect amendment

The architect recommends a Codex-first policy. The uploaded proposal showed that the strongest path is not a full UI first. It is a structured narrative engineering core:

```text
Story Repository first
Feature extraction second
Patch/PR third
IDE surface fourth
```

Therefore Page09 must prevent premature Narrative IDE work before the Codex-style core exists.

## 5. Chief Principal Compiler Engineer review

### 5.1 Compiler position

The compiler engineer argues that Page09 must be machine-readable. A prose-only feature taxonomy cannot prevent drift.

Page09 must produce:

```text
CommercialFeatureRecord
FeatureToContractMapping
ProviderSandboxPolicy
WriterApprovalPolicy
CodexFirstPolicy
AdvisoryVsBlockingPolicy
```

### 5.2 Compiler risk

The key risk is ambiguous permission. A feature like rewrite, evaluate, suggest, patch, export, or share may look harmless, but each implies different authority.

The compiler engineer requires explicit authority fields:

```text
requires_provider
requires_context_packet
may_generate_candidate
may_write_manuscript
may_mutate_canon
requires_writer_approval
requires_sandbox
requires_release_gate
inherits_from
successor_page
```

### 5.3 Compiler requirement

Every feature class must map to one of these execution modes:

```text
DOCUMENT_ONLY
READ_ONLY_ANALYSIS
ADVISORY_REPORT
SANDBOX_CANDIDATE
AUTHOR_APPROVED_COMMIT
EXPORT_ONLY
PLUGIN_SANDBOX
LEARNING_AUDIT_ONLY
```

### 5.4 Compiler amendment

The compiler engineer requires a default-deny rule:

```text
If a feature does not have an explicit mapping, it is not implementable.
```

This rule must be encoded in the Page09 release gate.

## 6. Chief System Principal Engineer review

### 6.1 Principal position

The system principal engineer agrees with both experts but adds operational control. Page09 should not become a market survey document. It must be an admission-control system.

The key question is:

```text
What exactly is allowed to enter the V1700 roadmap, under which contract, and at which future page?
```

### 6.2 Principal risk register

| Risk | Impact | Mitigation |
|---|---|---|
| commercial feature bypasses Page08 inheritance | lineage break | require inherited branchpoint references |
| provider feature opens too early | uncontrolled generation | provider sandbox policy |
| rewrite becomes commit | author authority failure | writer approval policy |
| evaluation becomes hard literary judgment | false blocking | advisory vs blocking policy |
| StoryPatch starts before Story Repository | unstable patch target | Codex-first sequencing |
| Narrative IDE starts before core contracts | surface-driven architecture | IDE deferred until Page11/Page15 |
| plugin/learning enters too early | non-reproducible behavior | Page17-only gated path |

### 6.3 Principal requirement

Page09 must explicitly preserve Page08 warnings:

```text
Stage185 remains local-known, not hub official.
dev_protocol_v3.0 and workflow/preflight_guide_v1.1 need canonical repository path confirmation.
```

These warnings do not block Page09, but Page09 must carry them forward.

### 6.4 Principal next-direction recommendation

The system principal engineer recommends that Page10 be started only after Page09 produces a taxonomy and mapping manifest. Page10 should then implement Story Repository / Story Bible / Codex / Context Preview according to Page09 classifications.

## 7. Three-expert debate

### 7.1 Architect vs Compiler

The architect wants Page09 to define a product strategy. The compiler wants schema-level admission control. The consensus is:

```text
Page09 is both product taxonomy and compiler-readable admission control.
```

### 7.2 Compiler vs Principal

The compiler wants strict default-deny. The principal accepts this but allows future extension through explicit successor trace. The consensus is:

```text
unmapped feature = blocked
mapped feature = allowed only in declared execution mode
future feature = PLANNED but not implementable until mapped
```

### 7.3 Architect vs Principal

The architect wants Codex-first sequencing. The principal agrees but warns not to make Page09 implement the Codex core. The consensus is:

```text
Page09 declares Codex-first policy.
Page10~Page13 implement the Codex-style core later.
```

## 8. Final expert consensus

The three experts agree:

```text
Page09 is the Commercial Absorption Constitution.
It must inherit Page08 authority.
It must convert external product ideas into V1700-native contracts.
It must adopt Codex-first sequencing.
It must block unmapped features.
It must separate advisory outputs from blocking gates.
It must keep provider, write, canon mutation, learning, plugin, and collaboration features behind explicit policies.
```
